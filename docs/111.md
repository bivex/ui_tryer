🔍 Выявленные проблемы ElementAnalyzer'а
1. False Negatives в определении кликабельных элементов 🚨
Проблема: Функция isClickableElement() слишком restrictively определяет кликабельные элементы, полагаясь только на CSS cursor стили (pointer, grab, grabbing). При этом игнорируются семантические HTML элементы.
Где проявляется:
// src/domain/services/ElementAnalyzer.ts:361-390private static isClickableElement(styles: ComputedStyles, elementId?: string): boolean {  // Только cursor стили проверяются  if (styles.cursor === 'pointer') {    return true;  }  // ... остальная логика только на cursor'ах}
Последствия:
<button> элементы без explicit cursor: pointer не считаются кликабельными
<a href="..."> ссылки могут быть пропущены
Элементы с role="button" игнорируются
Результат: Кликабельные элементы малого размера не проверяются на минимальную площадь клика
2. Несоответствие между слоями архитектуры ⚠️
Проблема: Инфраструктурный слой (ElementInspector) и доменный слой (ElementAnalyzer) по-разному определяют кликабельность элементов.
ElementInspector (src/infrastructure/dom/ElementInspector.ts:589-599):
const isInteractive = this.hasClickHandler(element) ||                     ['button', 'input', 'select', 'textarea', 'a'].includes(tagName) ||                     element.getAttribute('role') === 'button' ||                     element.getAttribute('role') === 'link' ||                     // ... и другие семантические критерии
ElementAnalyzer: Только cursor стили.
Последствия: Элементы, которые инфраструктура считает inspectable (кликабельными), могут быть пропущены анализатором размеров.
3. Отсутствие семантической информации в доменном слое 📋
Проблема: ElementAnalyzer получает только computedStyles и не имеет доступа к:
tagName элемента
HTML атрибутам (href, type, role, onclick и т.д.)
ARIA ролям
Наличию обработчиков событий
Текущее API:
ElementAnalyzer.analyzeElement(  elementId: string,  selector: string,   boxModel: BoxModel,  computedStyles: ComputedStyles,  // Только стили!  rules: DesignRules)
Последствия: Невозможно определить кликабельность на основе семантики HTML.
4. Потенциальные false positives в фильтрации 🎯
Проблема: Функции shouldSkipClickableAreaCheck() и shouldExcludeFromClickableCheck() могут быть слишком агрессивными в исключении элементов.
Примеры потенциальных проблем:
Элементы с классами .px-2.py-1 автоматически исключаются как "навигационные"
Маленькие элементы без "визуальной prominency" исключаются
Тонкие элементы (шириной < 24px при высоте > 100px) исключаются
5. Рекомендации по исправлению 💡
Вариант 1: Расширить isClickableElement
// Добавить семантическую информацию в параметрыprivate static isClickableElement(  styles: ComputedStyles,   elementId?: string,  tagName?: string,  attributes?: Record<string, string>): boolean {  // Проверить семантику  if (this.isSemanticallyClickable(tagName, attributes)) {    return true;  }  // Проверить cursor стили  // ...}
Вариант 2: Изменить архитектуру
Передать семантическую информацию из инфраструктурного слоя
Или сделать ElementAnalyzer менее restrictiven в определении кликабельности
Вариант 3: Улучшить фильтрацию
Сделать исключения более точными
Добавить больше критериев для определения "действительно проблемных" маленьких элементов
6. Тесты для проверки 🧪
Добавить тесты для:
Button элементов без cursor:pointer
Link элементов
Элементов с ARIA roles
Семантически кликабельных элементов малого размера
Приоритет исправления: Высокий - влияет на корректность обнаружения проблем UX с размерами кликабельных областей.
Ожидаемый результат: ElementAnalyzer должен корректно определять кликабельные элементы независимо от наличия explicit cursor стилизации.
